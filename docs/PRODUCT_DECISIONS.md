# Product Decisions

Decision · rationale · date.

- **Desktop landing scroll clamped to the zone below the fixed nav (2026-07-06).** The desktop "theater" used the document as its scroller, so the scrollable content's box started at y=0 behind the translucent banner — panel text visibly slid up under the glass nav mid-scroll, and the root scroller could rubber-band past both ends on Mac/iOS (Safari ignores `overscroll-behavior` on the root). Fix: `.lp-scroller` is now a real scroll container fixed between the nav's bottom edge (`--lp-nav-h`, kept exact via ResizeObserver) and the viewport bottom, with `overscroll-behavior:none` — content clips exactly at the banner line and the range hard-locks at both ends (the page itself is one viewport tall and never scrolls). Panels fill 100% of the container; `useScroll`/IntersectionObserver read the container. Keyboard operability preserved (WCAG 2.1.1): the region is focusable (`tabIndex=0`, axe scrollable-region-focusable) and document-level scroll keys (Space/arrows/PageUp-Down/Home/End when nothing is focused) plus wheel-over-nav are forwarded to the container. The MOBILE dot-dissolve path was deliberately left on document scroll — its visible content already lives in a fixed stage below the nav, its scroll bounds are pixel-exact, and its engine's svh/baseVH viewport fix (2026-07-05) depends on the document-scroll architecture.
- **Email + password login shipped alongside Google, via one combined Log in/Sign up modal (2026-07-05).** Replaces the old single "Continue with Google" button. Modal has tabs (Log in/Sign up), email+password fields, a divider, then Google — matches a reference pattern the founder liked (Admit.org), restyled to Marro's dark brand tokens. Forgot-password flow (request-reset + reset-password screens, PKCE-recovery-link detection so a reset link can never silently sign someone into the app) shipped in the same batch. Branded HTML email templates for Supabase's confirm-signup and reset-password emails are in `supabase/email_templates.md`, ready to paste into the dashboard. **Apple Sign-In stays deferred** (blocked on a paid Apple Developer account) — logged in `FUTURE_WORK.md`.
- **"This email already has an account — sign in instead" is APPROVED but NOT YET BUILT (2026-07-05).** Opus produced a full plan: detect via Supabase's `signUp()` returning an empty `identities` array for an existing confirmed email, show one neutral notice with both a "Sign in instead" link AND the Google button (can't distinguish password vs. Google-only accounts client-side, so offer both paths rather than guess). Security-reviewed as safe (minor, acceptable enumeration exposure for a budgeting app; no account-takeover path). Small scope (~half a day, 3 files + a CSS token). **Next session: implement this.**
- **New sending domain (joinmarro.com via Resend) needed real deliverability debugging, not just DNS setup (2026-07-05).** Root cause of two separate symptoms: (1) a duplicate/conflicting DMARC record (one auto-added, one manual) was invalidating DMARC entirely — fixed by removing the duplicate; (2) Hotmail spam-filtering and a Cornell inbox showing nothing at all are both just new-domain reputation effects, confirmed via Resend's delivery logs (status: Delivered, not Bounced) — nothing left to fix on our end. Cornell's issue needs their IT to run a message trace (ask sent, response pending).
- **Two founders now building in parallel — git workflow + Claude-awareness set up (2026-07-05).** Branch-per-person convention (`name/feature`) + actual PR review (previously bypassed) documented in `CONTRIBUTING.md` in plain language for non-coding founders. A shared, committed `PreToolUse` hook (`.claude/settings.json`) runs `gh pr list` before any commit/push so both Claude sessions notice the other founder's open PRs without needing to be told.
- **Mobile landing scroll: fixed a real Safari-specific bug, not just "feel" (2026-07-05).** After several rounds of scroll/animation polish (overscroll bounce, per-scene scroll distance, crossfade smoothing), the user's own hypothesis nailed the remaining ghosting/stuck-dust bug: the animation's scroll-progress math was re-reading live viewport height on every `resize` event — including the resize iOS Safari fires when its chrome (URL bar/toolbar) collapses/expands mid-scroll, unrelated to actual scroll distance. Fixed by locking a stable reference height that only updates on a real width change (rotation) plus switching to a chrome-immune `svh` CSS unit. That fix exposed a second bug: a perpetual goal-oscillation in the "rest-resolve" logic (the mechanism that finishes a stuck mid-transition into legible text) — for scroll positions landing in a specific band of a section, the resolve target and the raw scroll target volleyed back and forth forever, each flip yanking the settled text's opacity/position by a small amount, forever (never slept, never stopped) — read as a continuous jitter, worst on 60Hz phones. Fixed by making rest-resolve a one-shot latch that only real further scrolling can break, not the engine's own easing motion. Verified bit-for-bit static at rest across 90+ frames in preview; **still needs on-device iPhone confirmation**, same as the other scroll fixes today.

- **Roadmap reordered around validation; app simplifies to 3 tabs; weekly logging → monthly balance check-in (2026-07-01, full strategic audit with founder).** New ROADMAP.md supersedes the old phase order (old detail archived in `docs/HISTORY.md`). Load-bearing calls: **(1) Hide, don't delete** — Weekly/Savings/Subscriptions/Charts tabs hidden behind a revivable flag; subscriptions fold into Budget as fixed costs; everything returns later if users ask. **(2) The wedge ships before AI** — loan snapshot + debt-at-graduation projection + disbursement runway are the differentiator; everything built so far is the commodity layer. **(3) Ask the balance, not the expenses** — retention model shifts from weekly manual logging (discipline test, churn-by-shame) to a monthly "what's your checking balance?" check-in (one number off their banking app; a missed month self-heals since the next reading covers a longer span) + a zero-work weekly digest email (presence weekly, work monthly). **(4) A written beta gate** — <~1/3 of non-friend beta users completing the 2nd check-in after two fix cycles = stop building, rethink the loop. **(5) AI leads with the digest**, not anomaly detection (sparse monthly data makes anomaly output banal; banal AI erodes trust); BYOK + Jarvis voice mode + mascot cut. **(6) Family/partner support becomes an explicit onboarding money-in question** (feeds runway; the buried per-year otherIncome field wasn't findable). **(7) Docs discipline:** ROADMAP stays short (what's next), history/verification logs go to HISTORY.md, decision entries stay brief — the old narrative-log style was measurably bloating every session's context window.
- **Test harness = Vitest, colocated `*.test.js`, priority on the merge engine + money math (2026-07-01).** Phase 3.5 "test harness" half shipped: `vitest` + `jsdom` dev deps, a `test` block in `vite.config.js` (jsdom env — importing `src/lib/data.js` runs a module-scope `setInterval` whose callback references `window`), tests colocated next to source (`src/lib/*.test.js`). First suites target exactly the two surfaces CLAUDE.md flags as test-mandatory — the 3-way sync/merge engine and money/date math — via round-trip invariants (`applyChanges(base, diffStates(base,cur))` reproduces `cur`) rather than brittle snapshot-of-output. 62 tests green, build parity kept. Sentry (the error-monitoring half) deferred to a follow-up — needs an external project/DSN and touches the deploy path.
- **`daysUntil` off-by-one fixed — bug surfaced while writing tests (2026-07-01).** `daysUntil(dateStr)` compared a **noon-anchored target** (`dateStr+"T12:00:00"`, chosen for DST safety) against a **`now` floored to midnight** (`setHours(0,0,0,0)`), so every diff carried a spurious +0.5 that `Math.round` pushed up: today→1, tomorrow→2. Consequences (all in Subscriptions): renewal "due" detection (`daysUntil<=0`) fired a **day late**, the "renews in N days" banner read one high on the actual renewal day, and the ≤14/≤30-day windows were shifted. Fix = anchor `now` at noon too (`setHours(12,0,0,0)`) so both ends match and day counts are exact + DST-safe. Locked by a regression test (`daysUntil('today') === 0`). Rationale for fixing rather than test-around: it's an unambiguous correctness bug in user-facing countdown math, exactly what the harness exists to catch.
- **Tab form state resets on tab-switch (Phase 3.5 Step 2 Stage 2, 2026-06-30).** Extracting each tab into its own `src/tabs/*Tab.jsx` component means a tab unmounts when you leave it, so its private `useState` resets on return. This was a deliberate, per-tab call — for transient *forms* it's the expected/cleaner behavior (an in-progress entry doesn't silently linger under another tab): **Subscriptions** add/edit form, **Weekly** log-expense form + week-picker + CSV importer, **Savings** deposit/add-goal modals + projector APY, **Charts** view selections (pie month/range, trend + comparison toggles, hover), **Budget** category drag + add/remove-category modals — all reset on leave. **Deliberately kept shared (in `AppContext`, so they persist across tab switches — no behavior change):** `selMonth` (it also drives the always-visible header metrics, not just the Budget tab), `viewWeek` (browsing an archived week must survive a tab switch), and the add-category form fields (`newCat*`/`iconPickOpen`, shared between the Budget modal and the Categories tab). Rationale: real data is always persisted (Supabase + localStorage); only ephemeral form state is affected, and resetting an abandoned form is the safer default. Values whose persistence actually matters were the ones held in context.
- **App updates apply silently, never via a prompt (2026-06-30).** When a new build deploys, the service worker precaches it and waits; `SilentUpdater` (`src/main.jsx`) swaps to it **without any user-facing toast or reload prompt** — but only at a provably safe moment: the tab is backgrounded (`visibilityState==="hidden"`) AND nothing is mid-flow (no `role="dialog"` modal open, no focused `input/textarea/contenteditable`). If that moment never arrives, the waiting worker activates on its own on the next cold start. Rationale (user call): a prompt is itself a small interruption; the app's real data is continuously persisted (Supabase + localStorage), so the only thing a background reload can cost is ephemeral UI state, and the safety gates + cold-start fallback guarantee the user never sees a reload happen under an action. This is the concrete implementation of the roadmap's "auto-refresh at a safe moment, never mid-edit." (Considered + rejected: a visible "Update ready" toast with a Reload button — correct but adds a nudge the user didn't want.)
- **AI cost + model-routing strategy captured in `docs/AI_COST_MODEL.md` (2026-06-29).** Full plain-language model, but the load-bearing decisions: **(1) Routing** — Haiku by default → Sonnet for user-facing prose / multi-source synthesis → Opus only when a wrong answer is financially harmful or needs real multi-step reasoning (Haiku reads & sorts · Sonnet writes & explains · Opus reasons & advises). Cheap because the app already does the math and the AI mostly *narrates* a number we computed. **(2) Pricing shifts to staged all-paid:** free-for-everyone first (Stage 1, grow + prove value), then long-term **2-wk trial → $0.99 first month → ~$5/mo** with **no permanent free tier** — this supersedes the `STRATEGY.md` §4 "AI in every tier, free bounded forever" framing (now reconciled there). Grandfather early adopters at switch-on. **(3) Safeguards are AI-launch prerequisites** (treated like accessibility): hard spending cap + billing alerts + per-user rate limits go in *before* any AI ships — the only thing that can really hurt a self-funded founder is a runaway bill (bug loop / abuse), most acute in the free phase with no revenue backstop. **(4) Startup credits:** apply when flipping on Phase 4a so the clock starts with usage, not before. **(5) Model switching is auto-detect + human-approve** (dashboard flags, human tests & approves; never auto-swap the advice route untested). **(6) Also:** shrink images before sending, bundle habits+goals+savings into one digest call, one-shot (not chatty) voice/analysis, quality monitoring (thumbs + spot-check advice) alongside cost monitoring, AI must fail gracefully without breaking core budgeting, don't cheap out where wrong = harmful. Cost shape: typical user ~$0.15–0.40/mo, heavy <~$0.55/mo, ~6% of a $5 price (payment fees cost more than AI). **All numbers are estimates — first Phase-4a task is to measure real usage on a few users before trusting them.** Open: final free/trial caps, per-user limits, BYOK transport, anomaly sensitivity, benchmarking-density vs paywall tension.
- **Data ethics & monetization rules are now a binding doc: `docs/DATA_ETHICS.md` + CLAUDE.md rule 10.** Single source of truth for sell-vs-don't-sell, aggregate data, partner offers, consent, and all privacy copy; checked on every data/monetization/privacy-copy change; UI copy must match `privacy.html`. Core: two lanes — individual records (never sold/shared, RLS-enforced) vs true aggregates (group math, may be used, silent in UX, disclosed generically in policy). Never say "never sold" → "never sell your **personal info**"; never "anonymized" in a promise → "aggregate"; never mechanism words ("aggregate"/"group stats") in user-facing copy → benefit language. Partner offers: always shown, always clearly labeled "Partner/Sponsored," never framed as Marro's advice. Benchmarking = reciprocity unlock ("add your numbers to see how you compare"), not a consent wall (Glassdoor/Levels.fyi model — people opt in to get the payoff). True aggregates often aren't "personal data" → frequently need no consent at all. (2026-06-28)
- **Loan-capture model (Phase-5 design, locked 2026-06-28): capture the current balance *by type*, infer every rate that's public, ask only for what no rulebook can tell us.** A single "total debt" number is too lossy — med debt is layered (Direct Unsubsidized, Grad PLUS, sometimes private/school) at different fixed rates, and federal-vs-other also decides PSLF/IDR eligibility. BUT the fix is NOT a per-loan form: **federal rates are public + borrowing is structured (max Unsub then Grad PLUS up to COA), so we infer the federal rate mix from total + borrowing years** — no rate questions. The only unpredictable rates are non-federal, so we ask one toggle **"anything besides standard federal (Direct) loans?"** (the real line is "a rate we can look up" vs "a rate you give us" — catches school/institutional AND private, since school/institutional loans aren't federal-Direct and have no public rate table). "Yes" → one blended non-federal bucket (balance + rate), optional "+ add another" for multiple; full itemization lives in the Loans tab. Accuracy tiers shown honestly: total-only = *estimate*, +federal/private split = *good*, full per-loan = *precise*. (2026-06-28)
- **Offered ≠ Accepted ≠ Disbursed — never conflate (2026-06-28).** An aid award letter lists *offered* loan amounts (eligibility); the student *accepts* all/some/none; only *disbursed* is real debt. So the aid-letter scanner (Phase 4) treats every loan line as **offered** and confirms acceptance — never auto-adds an offer to debt. Two data sources, two jobs: **aid letter = plans one year's budget** (offered/accepted that year); **loan snapshot / StudentAid.gov = the running real debt** (disbursed, the source of truth for projections). Put the offered/accepted/disbursed status field in the loan data model from v1 so the scanner drops in cleanly later.
- **Loan snapshot is fused into the loan DATA layer, NOT a standalone pre-3.5 step (2026-06-28).** Run-through finding: building the snapshot + "Estimate" badge before any feature consumes loan numbers = plumbing with no fixture (badge has nothing to guard, can't validate end-to-end) AND bloats `index.html` right before the Vite migration. So: snapshot ships *with* its first consumer (Loans tab / a debt display), on the post-Vite foundation, with tests. Asked at setup but **skippable**; incomplete → calm **"Estimate — add your loans to make this exact"** badge on loan-dependent numbers (NOT an alarming "inaccurate" warning — rule 9 + ADA), post-first-run reminder with snooze (later/next time/never, persisted in `app_state`), one-tap to open anytime; complete → no badge/reminders. Loans tab: user-named loans, editable, feeds the Aid/Detail tab. Sequencing rule: **loan DATA layer → loan-aware AI** is the only hard dependency; 4a (budget-only AI) stays early and independent. (See ROADMAP "Sequencing decision 2026-06-28".)
- **Onboarding "joined partway through med school" fix = ask the start year, not the M-year (2026-06-28).** "Which year are you in (M1–M4)?" secretly requires knowing each school's academic-calendar boundaries and is ambiguous in summer. Instead ask the one unambiguous fact: **"When did you start? → Fall [year]"** (default current fall), which feeds `generateYearConfigs(startYear, len)` directly with no calendar knowledge. Exact term dates stay the editable heuristic (the academic-calendar auto-fetch remains the separate FUTURE_WORK item). Also this batch: MD-PhD suggested length **8 → 7**; year count becomes a **stepper** (suggested + ±/editable) not a number-button row; remove the dead `!firstRun` redo copy (redo-setup button was removed June 20). Keep the avatar step ("fun in a moment of seriousness" — user call). (2026-06-28)
- **Aid-year delete = soft delete (2026-06-28).** Removing a year in the Aid tab archives its data (grant/tuition/housing/inputs) instead of discarding it, so reinstating the year restores the old numbers exactly. Optional immediate "Undo" toast on top.
- **Account deletion + data export added as a pre-launch must (2026-06-28).** Self-serve "delete everything" + "download my data" — legally expected (GDPR/CCPA), app-store requirement, and the biggest single trust lever for skeptical med students.

- **All CDN `<script>` deps are version-pinned — never use an unpinned/`@latest` URL.** `@babel/standalone` was loaded unpinned; unpkg auto-upgraded it to 8.0.2, whose preset-react defaults to the automatic JSX runtime and injects a top-level `import` into the compiled output. Our app script is a classic `text/babel` script (not a module), so that import is a fatal parse error ("Cannot use import statement outside a module") — React never mounts and the static boot rings hang forever. Hit local + production identically. Pinned Babel to `7.29.7` (classic `React.createElement`, no imports). If migrating to Babel 8 later, must also force `{runtime:"classic"}` on preset-react or switch the script to a real module bundle. (2026-06-20)

## Phase 2.5b — Auth + Supabase (2026-06-13)
- **Hard login gate** — app requires Google sign-in; no anonymous/local-only mode. One code path, per-user data from day one. (2026-06-13)
- **Keep the blob model** — one jsonb `state` per user in `app_state`, not normalized tables. The 3-way merge engine is transport-agnostic, so Supabase only replaced the Gist transport (`gistFetch/gistWrite` → `stateFetch/stateWrite`); `save`/`checkAndPull`/`resolveConflict`/ConflictModal untouched. Normalizing is a Phase-6 concern, not now. (2026-06-13)
- **localStorage stays as offline cache + merge ancestor; no Realtime** — existing focus/30s polling is enough; Realtime would add a websocket dependency and failure modes for a tiny user base. (2026-06-13)
- **Anon (publishable) key hardcoded in the public repo** — security is RLS, not key secrecy; no build step means env vars aren't an option client-side anyway. Service-role key + Google client secret never in repo. (2026-06-13)
- **School via searchable picker + free-text Other** in a one-time, non-dismissable ProfileModal after first login — clean data for Phase 3/6 benchmarking; embedded curated US MD/DO list (expand to full AAMC/AACOM before public launch). (2026-06-13)
- **Single-file architecture survives 2.5b** — supabase-js loads as a CDN UMD global alongside React/Recharts; still no build step. (2026-06-13)
- **No school is special-cased — every school flows through the same generic generated year-config** (financial fields default to 0, filled later via Aid tab / future aid-letter scan). Supersedes the 2026-06-11 "WCM amounts stay as data" stance: hardcoded `YEAR_CONFIGS`/`DEFAULT_MONTHLY` and the Aid-tab WCM cost-of-attendance table were removed. Rationale: hardcoding any one school's numbers as a preset doesn't scale to a national audience. (2026-06-14)
- **Year dates come from a swappable provider, not a "predictive model" yet** — tier 1 ships now: arithmetic heuristic anchored ~Aug 1, leap-aware, fully editable. Predicting from prior years / scraping calendars needs a data source we don't have at onboarding (no per-school calendar API); that's the tier-2/3 future work. Budgeting only needs the ~12-month financial boundary, not day-precision, so the heuristic is sufficient now. (2026-06-14)
- **Calendar fetch will be LLM extraction (lazy + cached), not 200 bespoke scrapers** — when a user from an uncovered school onboards, fetch+extract that school's calendar once via Claude, cache per-school in Supabase, reuse for all future users. Same engine as the planned aid-letter scan. Both Phase 4 (needs the small backend). (2026-06-14)
- **Progressive setup: onboarding questions are versioned (`SETUP_VERSION` + `SETUP_STEPS`)** — new users answer inline; existing users behind a newly-added question get a focused glass popup for just that step, never a full re-onboarding. v1 grandfathers existing users (they keep their configured years). Rationale: lets us add setup questions over time without disrupting or nagging established users. (2026-06-14)
- **Program type (MD/DO/MD-PhD) is never asked — derive from school name** (LCME vs COCA lists). Deferred until Phase 4 has a consumer to avoid dead code. (2026-06-14)
- **"Key notes" (Customize tab) are derived from the user's own data, not static copy** — built from the active year's grant/tuition/health-ins/housing, the user's Step-exam goals + exam budget, and whether the plan has an extended year. Notes appear only when relevant (e.g. health-insurance note only if `healthIns>0`, extended-year note only if a year has `type==="extended"`) and restate the user's real numbers. Replaced the old hardcoded WCM block (NYC/$5,232 fees/grant-deduction copy). Extended-year Aid-tab banner also de-WCM'd ("costs and grant often differ — check with your aid office") since a generic school may not waive tuition. Rationale: the section should teach the user about *their* plan and stay correct for any school, updating as they fill the app. (2026-06-15)

## Phase 3 — program model (2026-06-15)
- **"Extended year" concept removed entirely** — it was a WCM special-case. Years are now plain numbered years (Year 1, Year 2, …). `generateYearConfigs(startYear, lengthYears)` dropped its 3rd `extendedYear` arg; year objects no longer carry a `type` field. Boot migration converts any legacy `type:"extended"` year into the next numbered year (data preserved, relabeled) and renumbers all years sequentially by start date. (2026-06-15)
- **Dual-degree support (MD-PhD / MD-Master's)** — onboarding step 4 ("Your program") now asks track after school pick: `<degree> only` / `<degree>-PhD` / `<degree> + Master's` / `Other dual degree`. PhD/Master's **field** and **granting institution** (same-as-school toggle or free-text) are optional ("you can add later in Settings"). Program length widened to 3–8 yrs, pre-filled by track (MD 4 / Master's 5 / PhD 8) and adjustable. Stored in the app-state blob as `data.program {degree,dual,phd{field,institution},masters{field,institution}}` — no Supabase schema change. (2026-06-15)
- **Degree (MD/DO) derived from school name, not stored per-entry** — every COCA school name contains "Osteopathic", no LCME school does, so `degreeForSchool(name)` is a regex; satisfies the old "MD/DO from school name" backlog item with zero list maintenance. (2026-06-15)
- **DO dual-degree options gated by a curated `DO_DUAL` map** (per user: DO/PhD & DO duals are rare).
