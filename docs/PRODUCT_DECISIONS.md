# Product Decisions

Decision · rationale · date.

- **Auth/invite flow redesigned around a single redemption point — 2026-07-08, branch `ethan/auth-flow-redesign` (cofounder-driven UX review).** The invite code now has exactly one home: the InviteGate. **(1) Removed the optional "Invite code" field from sign-up** — it duplicated the gate and its stash-at-signup path silently lost codes when email confirmation happened on a different device. **(2)** `?invite=` email links are stashed by AuthModal on mount (localStorage, with a `from` source tag), surviving all three URL-losing auth paths (OAuth redirect, password-sign-in hard reload, email-confirm round-trip); the gate reads URL-or-stash, auto-fills, auto-redeems, and now shows a visible congrats state ("You're off the waitlist!" vs "You're invited!" via the new `&from=waitlist` tag on waitlist-approval CTA links). App.jsx's silent background redeem was removed — failures are now always visible to the user at the gate. **(3) Failed sign-in offers "New to Marro? Create an account"** (email carried across) instead of auto-switching tabs — auto-detection of "no such account" was rejected deliberately: it requires exposing account-existence to anonymous visitors (email enumeration, the same leak `email_has_account` was locked to service_role to prevent). **(4)** Waitlist-approval links open the Log in tab (those users already have accounts); friend invites open Sign up.

- **Ambassador/admin console bugfix round — 2026-07-08, same branch (`claude/ambassador-admin-overhaul`), found in live founder testing after the first deploy.** **(1) Role status was leaking into the freeform member note** — `set_role`/`add_admin`'s durable-access grant wrote the literal string `'ambassador'`/`'admin'` into `allowed_emails.note` on first grant, so the Members list showed it as if an admin had typed it, and it never updated when the role changed. - **Ambassador roster + members management + two security fixes — built 2026-07-08 (branch `claude/ambassador-admin-overhaul`).** Extends the 2026-07-07 invite/waitlist/admin console. **(1) Two real security bugs found on audit, fixed:** C1 — `redeem_invite_code()`'s single-use claim guarded on `redeemed_by IS NULL`, but `redeemed_by` is `on delete set null`; if a redeemer deleted their account, their single-use code silently became re-claimable by a second person. Switched the guard (and every "is this code used" check, SQL + both frontend files) to `redeemed_at IS NULL`, which nothing ever nulls. C2 — deleting your own account left `admins`/`user_roles` rows (keyed by email, no FK) intact, so a deleted admin/ambassador's privilege silently reactivated if that email ever signed up again; `api/delete-account.js` now always wipes both, regardless of the choice below. **(2) Owner product choice: deleting your own account keeps your `allowed_emails` row.** Rejoining with the same email skips the invite gate — deleting shouldn't cost you your spot — but your data does NOT come back (new signup = new auth uid, app_state/profiles are really gone; "keep the gate open, not the data"). **(3) Admins + ambassadors get app access automatically** (`is_email_allowed()` now also checks `admins`/`user_roles.is_ambassador`), but granting either role ALSO upserts a durable `allowed_emails` row — otherwise later removing the role (`clear_role`/`remove_admin`) would silently revoke ALL app access as a side effect of what the UI frames as just losing a role, since access was computed live off the role table with nothing durable backing it. Revoking actual access is the separate, explicit `revoke_access` action (keep-data vs. delete-everything, admin-triggered version of the owner's own-account choice above). **(4) "Quota" renamed "invite limit"/"invites" everywhere user-facing;** admins get an effectively-unlimited limit; codes minted from the console (`issued_by_admin`) are excluded from personal invite-limit accounting on both the server and the client, so an admin's own referral count doesn't silently shrink every time they bulk-mint codes for the waitlist. **(5) New in-app notifications** (`user_notifications`, `dismiss_notification()` RPC) — a dismissible banner tells a member when someone they invited joins, their invite limit changes, they're granted ambassador/admin status, or their access changes; every admin action that affects a user writes one. **(6) Member/ambassador self-serve:** `revoke_own_code()` lets anyone revoke their own unused code (auto-refunds the slot — revoked codes were already excluded from the limit count) and a new `api/send-invite.js` endpoint lets a member email one of their own unused codes to someone via Resend (ownership + status re-checked server-side, 20-sends/24h rate limit — this endpoint is reachable by every member, not just admins). Caught in review before merge and fixed: a Members-section note field was writing to `user_roles.note` instead of `allowed_emails.note` (silently didn't persist, and spuriously made plain members show up as ambassadors, since the ambassadors list had no `is_ambassador` filter) — split into a dedicated `set_member_note` action. **Verified:** `npm run build` clean, all `api/*.js` syntax-checked and import cleanly under Node (fixed a latent issue where `api/*.js` imported `src/lib/data.js`, dragging a browser-only `window`-referencing `setInterval` into the serverless runtime — now a side-effect-free `api/_config.js`). **Not yet live-tested end-to-end** — needs `supabase/notifications.sql` + `supabase/invites_waitlist.sql` + `supabase/allowed_emails.sql` re-run in Supabase Studio, a `RESEND_API_KEY` Vercel env var, and a real signed-in admin session.
- **Invite codes + waitlist + admin console — built 2026-07-07 (branch `claude/invite-codes-waitlist-admin`).** Load-bearing calls, mostly locked in the spec + an Opus design review before building: **(1) Gate keeps the session alive.** The old gate signed a non-allowlisted user out immediately; redemption RPCs read `auth.uid()`/`auth.jwt()->>'email'`, so they must stay signed in. A non-allowlisted user now lands on `<InviteGate>` with the session intact — safe because every data table is RLS'd to their own `user_id` and nothing loads until they're allowed; they only sign out via an explicit "Back to sign in." A successful redeem bumps a `gateNonce` that re-runs the existing boot effect (re-checks `isEmailAllowed()`, now true) — no full page reload. **(2) Atomic single-use redemption (no TOCTOU race).** `redeem_invite_code()` claims a code with a single `UPDATE … WHERE redeemed_by IS NULL AND revoked_at IS NULL RETURNING`, so two concurrent redeemers can never both win one code. **(3) Deliberate error granularity.** A *used* code returns the SPECIFIC "already used — ask your inviter for a fresh one" (so the redeemer knows to go back to whoever invited them); every other failure is generic ("that code isn't valid"). This intentionally leaks that a used code exists, mitigated by (4). **(4) Brute-force lockout.** The codes are short/human-typeable, so every failed attempt is logged to `invite_attempts` and ≥10 failures/hour per user returns `locked` — this also throttles enumeration of the used-code signal from (3). **(5) Quota is not client-writable.** 5/regular, 15/ambassador, enforced server-side inside the SECURITY DEFINER `generate_invite_code()` reading a `user_roles` table users can only SELECT (never UPDATE) — otherwise a user could inflate their own quota. Revoked codes free the slot back up. **(6) Explicit RLS on every new table** (the `ensure_rls` trigger enables RLS but leaves tables deny-all until policies exist — CLAUDE.md rule 4); `admins`/`invite_attempts` get no client policies at all (SECURITY-DEFINER / service-role only). **(7) Admin is bootstrapped, then self-managed.** Seed one owner (`jawadhijazi7@gmail.com`); an existing admin grants/revokes others from the console (the founder's answer: "the console can give access to certain accounts"). `is_admin()` only decides console *visibility* — `api/admin.js` re-verifies admin server-side on every action (same token-verification trust boundary as `api/delete-account.js`), so the client flag is never the security border. **Verified:** `npm run build` (whole module graph incl. the new lazy AdminTab chunk), eslint clean, and a standalone in-preview render of `<InviteGate>` (both modes, generic-error `role="alert"` path, mobile). **Not yet live-tested end-to-end** — needs the SQL applied in Supabase Studio + a real signed-in session (Google OAuth can't complete in the preview pane); AdminTab + the referral modal are behind the admin/signed-in gates and will be verified in the owner's own Chrome.

- **Mobile landing scroll: fixed a real Safari-specific bug, not just "feel" (2026-07-05).** After several rounds of scroll/animation polish (overscroll bounce, per-scene scroll distance, crossfade smoothing), the user's own hypothesis nailed the remaining ghosting/stuck-dust bug: the animation's scroll-progress math was re-reading live viewport height on every `resize` event — including the resize iOS Safari fires when its chrome (URL bar/toolbar) collapses/expands mid-scroll, unrelated to actual scroll distance. Fixed by locking a stable reference height that only updates on a real width change (rotation) plus switching to a chrome-immune `svh` CSS unit. Fixed by making rest-resolve a one-shot latch that only real further scrolling can break, not the engine's own easing motion. Verified bit-for-bit static at rest across 90+ frames in preview; **still needs on-device iPhone confirmation**, same as the other scroll fixes today.

- **`daysUntil` off-by-one fixed — bug surfaced while writing tests (2026-07-01).** `daysUntil(dateStr)` compared a **noon-anchored target** (`dateStr+"T12:00:00"`, chosen for DST safety) against a **`now` floored to midnight** (`setHours(0,0,0,0)`), so every diff carried a spurious +0.5 that `Math.round` pushed up: today→1, tomorrow→2. Consequences (all in Subscriptions): renewal "due" detection (`daysUntil<=0`) fired a **day late**, the "renews in N days" banner read one high on the actual renewal day, and the ≤14/≤30-day windows were shifted. Fix = anchor `now` at noon too (`setHours(12,0,0,0)`) so both ends match and day counts are exact + DST-safe. Locked by a regression test (`daysUntil('today') === 0`). Rationale for fixing rather than test-around: it's an unambiguous correctness bug in user-facing countdown math, exactly what the harness exists to catch.
- **Tab form state resets on tab-switch (Phase 3.5 Step 2 Stage 2, 2026-06-30).** Extracting each tab into its own `src/tabs/*Tab.jsx` component means a tab unmounts when you leave it, so its private `useState` resets on return. - **App updates apply silently, never via a prompt (2026-06-30).** When a new build deploys, the service worker precaches it and waits; `SilentUpdater` (`src/main.jsx`) swaps to it **without any user-facing toast or reload prompt** — but only at a provably safe moment: the tab is backgrounded (`visibilityState==="hidden"`) AND nothing is mid-flow (no `role="dialog"` modal open, no focused `input/textarea/contenteditable`). If that moment never arrives, the waiting worker activates on its own on the next cold start. Rationale (user call): a prompt is itself a small interruption; the app's real data is continuously persisted (Supabase + localStorage), so the only thing a background reload can cost is ephemeral UI state, and the safety gates + cold-start fallback guarantee the user never sees a reload happen under an action. This is the concrete implementation of the roadmap's "auto-refresh at a safe moment, never mid-edit." (Considered + rejected: a visible "Update ready" toast with a Reload button — correct but adds a nudge the user didn't want.)

## Usage/event logging (2026-07-06)
