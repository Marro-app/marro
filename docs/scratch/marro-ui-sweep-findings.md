## Area 1 вЂ” Header & global chrome

### Findings

1. **[MEDIUM] Modal focus-on-open ignores an input's `autoFocus`, always lands on the first focusable element (the вњ• close button) instead.**
   Repro: Settings в†’ "Delete my account" в†’ focus lands on "Close dialog" (вњ•), not the `#delete-confirm-input` text field, even though that input has `autoFocus` in source (`src/App.jsx:1013`). Root cause confirmed in `src/components/primitives.jsx` `Modal` component (~line 233): its mount effect does `(focusables()[0] || panel)?.focus()`, which runs after React's own `autoFocus` and always wins, grabbing focus onto whatever is first in DOM order (almost always the вњ• button) instead of respecting a deliberately-placed `autoFocus` on a later field. Affects every modal in the app that relies on `autoFocus` on a non-first control (Delete-account confirm being the most consequential, since it's a destructive-action dialog where keyboard users would expect to land in the confirmation field). Given the repo's ADA/keyboard-accessibility priority (CLAUDE.md rule 7), this is worth a real fix (e.g. Modal should prefer an element carrying `autoFocus` if one exists, falling back to the first focusable).

2. **[MEDIUM] Focus is not restored to a sensible element after closing a modal opened from the Settings menu.**
   Repro: Settings в†’ "Delete my account" в†’ Cancel в†’ `document.activeElement` is `<body>`, not the Settings gear button or any menu item. Root cause: the trigger buttons in the Settings dropdown call `setSettingsOpen(false)` in the same click handler that opens the modal, so by the time the Modal's cleanup effect tries `prevFocus.focus()` the previously-focused element (the menu item) has already unmounted вЂ” the call silently no-ops and keyboard focus is lost to `<body>`. Reproducible for any Settings-menu-triggered modal (Delete account, Export, Categories), not just one. Keyboard-only users lose their place after closing any of these.

3. **[MEDIUM] No upper bound on Quick Add "Amount" вЂ” a 9-digit expense ($999,999,999) is accepted and breaks the Plan-vs-actual chart's number formatting.**
   Repro: Quick add в†’ Food & groceries в†’ Amount `999999999` в†’ Add expense. The "Add expense" button is enabled (only a `parseFloat(amt) > 0` check exists in `src/components/modals.jsx:83`, no ceiling). After adding, the Plan-vs-actual bar chart's Y-axis renders raw unformatted numbers (`$300000000`, `$600000000`, вЂ¦) and the "Notes" total shows `$1200000000` with no thousands separators вЂ” a jarring, clearly-unintended visual break. Recommend a sane input ceiling (e.g. reject/clamp above some six-figure threshold) plus defensive number formatting on the chart axis regardless.

## Area 2 вЂ” Budget tab

### Findings

1. **[HIGH вЂ” a11y] Category drag-reorder has zero keyboard equivalent; the в ї handles aren't even focusable.**
   Repro: `src/tabs/BudgetTab.jsx` line ~102-108 вЂ” the reorderable row is a `<div draggable={!isAuto} onDragStart/onDragOver/onDrop>` with a `<span>в ї</span>` handle. Verified via JS: every handle span has `tabIndex:-1`, no `role`, and no `onkeydown`; the parent `div[draggable]` also has no `tabIndex` and no keydown handler. `reorderCats(fromId, toId)` (`src/App.jsx:852`) is only ever called from the `onDrop` handler. A keyboard-only user cannot reorder budget categories at all вЂ” not "hard to use," genuinely impossible, since the handle can't even receive focus via Tab. Given the repo's WCAG 2.1 AA / "every interactive element keyboard-reachable" rule (CLAUDE.md rule 7), this is a real gap: needs a focusable handle (or the row itself) with arrow-key move-up/move-down support calling `reorderCats`.

2. **[LOW] Subscription delete has no confirmation step, unlike category removal.**
   Repro: Budget tab в†’ "Manage" (fixed monthly costs) в†’ "Delete" next to a subscription row в†’ the subscription is removed **immediately**, no "are you sure?" step. By contrast, removing a budget category on the same tab (the "вњ• Remove" button next to each category row) opens a confirm modal ("Remove `<category>` from `<month>`? You can add it back anytime.") before acting. The inconsistency means users get a safety net for one destructive action but not the sibling one, on the same screen. Not CRITICAL since subscriptions can be re-added, but worth aligning with the category-removal pattern for consistency (CLAUDE.md rule 9 UI-consistency intent).

3. **[LOW] No way to rename a category anywhere in the app, including Budget tab.**
   Repro: grepped the whole `src/` tree for any input bound to `cat.label` or a `setLabel`/`renameCat` function вЂ” none exists. `CustomizeTab.jsx` (the other place categories are managed) only lets you change a category's **icon**, not its name; `addCat()` (`src/App.jsx:843`) sets `label` once at creation and it is never editable again short of delete-and-recreate. Not a Budget-tab-specific bug (category rename was never built anywhere), but worth flagging since the test plan explicitly expected inline rename to be possible on this tab.

1. **[MEDIUM] No validation (no `min=0`) on any numeric field lets negative values in, and negative "Housing (monthly)" silently corrupts the Budget tab's shared housing line and the header MONTHLY PLAN tile.**
   Repro: Aid & Detail в†’ Year 1 в†’ set "Housing (monthly)" to `-500` (native-setter `input`/`change` events, field is a plain `<input type="number">` with no `min`, `src/tabs/AidTab.jsx:76-77`). 2. **[LOW] Negative "Total aid (annual)" is mislabeled as "TBD" in the 4-year overview table instead of showing the actual (negative) number.**

3. **[LOW] Editing a year's start/end date does not update that year's `label` вЂ” the label can become stale/misleading relative to the actual date range shown right next to it.**

Not CRITICAL вЂ” doesn't break any math вЂ” but is a real, easily-triggered visual inconsistency between two adjacent pieces of UI showing the same year.

4. **[LOW] The `DateField` picker has no explicit "Clear" affordance.**
   Repro: read `src/components/pickers.jsx` (`DateField`, lines 80-139) and confirmed live in the popover DOM вЂ” the footer only ever renders a single "Today" action (line 133); there is no "Clear"/"None" button anywhere in the component. 1. **[HIGH] Quick Add modal does not restore focus to its trigger button on close (Escape or вњ•) вЂ” focus is lost to `<body>`.** Repro: focus/click "Quick add expense" в†’ modal opens (focus correctly force-lands on the вњ•, see item 3 below) в†’ press Escape (or click вњ•) в†’ `document.activeElement` is `<body>`.
